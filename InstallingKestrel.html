Note: Kestrel is a prototype middleware being tested by STAR to access the Clemson campus grid, it is not a production middleware approved by OSG.

Installing a Kestrel Worker Node

Kestrel is a new scheduler under development at Clemson University that uses XMPP as a message passing interface. A paper describing its architecture can be found here. Note that Kestrel is a prototype for the next generation of cloud computing batch schedulers and is not a production system.

To install Kestrel and connect to the Clemson University pool, you will need Python >= 2.6 but < 3.0. If Python 2.6 isn't the default version of python for your system, you can download and install it. After you run make pay attention to the list of modules that failed to build, as this depends on which libraries are available on the system. Of particular interest is the ssl module, as this is necessary for Kestrel. If any needed module fails to build, install the relevant OS package (generally suffixed with -dev or -devel) and run make again. 

wget http://www.python.org/ftp/python/2.6.5/Python-2.6.5.tar.bz2
tar xjf Python-2.6.5.tar.bz2
cd Python-2.6.5
./configure
make
make altinstall

Next we will need to install the Python setuptools package. This provides a Python package manager that is needed to install several of Kestrel's dependencies. If Python >= 2.6 is the default Python on your system, you may be able to use a package provided by your distribution for this step. 

cd .. # if you were still in the Python-2.6.5 directory
wget http://pypi.python.org/packages/source/s/setuptools/setuptools-0.6c11.tar.gz
tar xzf setuptools-0.6c11.tar.gz
cd setuptools-0.6c11
python2.6 setup.py build
python2.6 setup.py install

Now we'll install those additional packages. OPTIONAL: This step is only necessary if you want to install Kestrel in a self-contained environment. 

cd .. # if you were still in the setuptools-0.6c11 directory
easy-install-2.6 virtualenv
mkdir kestrel
cd kestrel
virtualenv .
source bin/activate

Now we'll actually get the Kestrel source and perform the setup. SleekXMPP is required for Kestrel to operate. 

wget http://github.com/fritzy/SleekXMPP/tarball/0.9RC1
tar xzf fritzy-SleekXMPP-0.9RC1-0-g935ee4d.tar.gz
cd fritzy-SleekXMPP-67c6f50
python2.6 setup.py install

Matplotlib is required to use the log analyzer. It is otherwise optional 

easy_install-2.6 numpy
easy_install-2.6 matplotlib

Now, to install Kestrel itself. 

easy_install-2.6 kestrel

You'll need to create a config file at ~/.kestrelrc 

# XMPP
# ---------------------------------------------------------------------
# Credentials for connecting to an XMPP server.
# Options:
#   method: client|component - The type of XMPP connection to use.
#                              Accepted values are client to use a
#                              typical XMPP client session, or
#                              component to use a more scalable
#                              server component session.
#   jid: string              - The username or component name
#   password: string         - This is plaintext storage, modify read
#                              permissions of this file accordingly.
#   server: string           - Optional, specify the name of the XMPP
#                              server. Normally, the JID will provide
#                              the server name.
#   port: int                - Optional, specify the connection port.
#                              The default is 5222.

# NOTE: For the Clemson pool, workers should have JIDs of the form:
#       worker_X@kestrel.cs.clemson.edu where X is an integer from
#       1 to 10,000. For use with many concurrent VMs, you will need
#       to make a script to generate an appropriate config file with
#       a random worker JID.
[XMPP]
method=client
jid=worker_1@kestrel.cs.clemson.edu
server=kestrel.cs.clemson.edu
password=kestrel

# Worker
# ---------------------------------------------------------------------
# Configuration for running a Kestrel worker agent.
# Options:
#  manager: string - The JID for the manager agent that the worker
#                    will accept tasks from.
#  profile: string - A space separated list of tags that identify
#                    the capabilities of the worker. The list
#                    can span multiple lines so long as the first
#                    character of the line is a space.
[worker]
manager=manager.kestrel.cs.clemson.edu
profile=STAR

To run the worker client 

source $VIRTUALENV_DIR/bin/activate # only if you installed with virtualenv
easy_install-2.6 -U kestrel # Optional: to update kestrel
kestrel worker &

Likewise, use manager or analyzer as arguments to the kestrel executable in order to run those components. Kestrel logs to stdout by default, so output redirection is necessary to save the log file. The analyzer argument requires an additional argument which is the path to the log file to be analyzed.

Congratulations! The machine is now a worker node connected to the Clemson Kestrel cloud.

Using Kestrel

Before using Kestrel, it is advisable to install the Kestrel worker client and run it on a few machines. Otherwise, you may not be able to guarantee the availability of resources. Note: The Kestrel worker node client is not required to be installed on the submission machine. Only a compatible chat client such as Pidgin, Adium, GTalk, etc. is required.

Preparing a Job

Write a shell script that will manage tasks.
  1. The script will be called with an integer index value from 0 to N-1 where N is the number of tasks for a job. Mapping this index to the actual desired outputs is done by this script.
  2. It is the script's responsibility to download and upload any input and output files.

Submitting a Job

Given a job shell script located at "~/runtask.sh" and a pool of running VMs to execute a job with 10,000 tasks, send the following message to manager.kestrel.cs.clemson.edu using Pidgin, Adium, GTalk, etc:

{'type':'job', 'command':"~/runtask.sh", 'cleanup':"~/cleanup.sh", 'requires':["foo", "bar", "baz"], 'queue':10000}

The cleanup script will be run after the task has been completed and before the next is accepted. This may be used to perform tasks that would potentially disrupt the machine's state (e.g. shutdown the machine). The requires attribute is a list of tags that determine whether or not a particular worker will accept the job. The requires list must be a subset of the provides list in the worker profile (defined in install_root/Kestrel/start.py) in order for the job to be accepted.

The manager will respond with either {'type':'job/approved', 'job_id': 483} or {'type':'job/rejected'} depending on whether or not the manager accepted the job.

Checking Job Status

  1. Send the following message to manager.kestrel.cs.clemson.edu using Pidgin, Adium, GTalk, etc: {'type':'status'} The manager will respond with a list of your submitted jobs and their progress.
  2. To get the status of a particular job, say 317, use this format: {'type':'status', 'job_id': 317}

Canceling a Job

  1. To cancel all submitted jobs, send the following message to manager.kestrel.cs.clemson.edu: {'type':'cancel'}
  2. To cancel a particular job, say 428, use this format: {'type':'cancel', 'job_id': 428} Use the status command to find the job's id number.

Job Completion

Once all tasks have been completed (either successfully or with errors or from cancellation) for your job, say 78, the manager will send the following message to your XMPP client (Pidgin, Adium, GTalk, etc): {'type':'job/finished', 'job_id': 78}
